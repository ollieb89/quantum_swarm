## Gap Root Causes and Fixes

### Gap 1 — RISK-07 / RISK-08: InstitutionalGuard Not Wired Into Graph

**Location:** `src/graph/orchestrator.py`

**Root cause:** `institutional_guard_node` was registered (`workflow.add_node("institutional_guard", ...)` at line 249) but no `add_edge` or `add_conditional_edges` call was made to place it in the execution sequence. The graph compiles without error because LangGraph allows registered-but-disconnected nodes.

**Fix:** Add two edges in `orchestrator.py`:
```python
workflow.add_edge("claw_guard", "institutional_guard")
workflow.add_edge("institutional_guard", "data_fetcher")
```
(Or wire via conditional edge if rejection path is needed.)

**Requirements closed:** RISK-07, RISK-08, and downstream EXEC-04 `portfolio_risk_score` field.

---

### Gap 2 — MEM-06: Validation Gate Bypassed by Call Order in persist_rules()

**Location:** `src/agents/rule_generator.py` lines 106-122

**Root cause:** Phase 12 MC-01 fix added `update_status(rule.id, "active")` inside the `for rule in rules:` loop to ensure generated rules become active. Phase 10 added a `RuleValidator.validate_proposed_rules()` call after the loop. The Phase 12 fix runs first: all rules are promoted to active and saved to disk before the validator runs. The validator reloads from disk, finds no proposed rules, and no-ops.

**Fix:** Restructure `persist_rules()` to:
1. Add all rules as `proposed` (existing line 107)
2. Call `RuleValidator.validate_proposed_rules()` — this promotes/rejects via `update_status()` inside the harness
3. Remove the explicit `update_status(rule.id, "active")` call at line 108 (the validator now handles promotion)

This satisfies both MC-01 (rules become active after persist) and MEM-06 (only rules passing the 2-of-3 harness become active).

**Requirements closed:** MEM-06
