# Phase 9 Context: Structured Memory Registry

Implementation decisions for transitioning to a governed, machine-readable rule system.

## 1. Core Schema
- **File**: `data/memory_registry.json` (Primary source of truth).
- **Format**:
    ```json
    {
      "rules": [
        {
          "id": "mem_0001",
          "title": "Widen stops in high-volatility trend regimes",
          "type": "risk_adjustment",
          "scope": "trade_entry",
          "condition": { "regime": "high_volatility_trend" },
          "action": { "parameter": "atr_multiplier", "from": 2.0, "to": 2.5 },
          "evidence": { "sample_size": 41, "confidence": "medium" },
          "status": "active",
          "version": 1,
          "created_at": "...",
          "updated_at": "..."
        }
      ]
    }
    ```

## 2. Lifecycle Controls
- **Statuses**:
    - `proposed`: Generated by RuleGenerator, pending validation.
    - `active`: Validated and currently influencing live behavior.
    - `deprecated`: Replaced or no longer relevant.
    - `rejected`: Failed validation or manual override.
- **Versioning**: Each update increments the `version` field.

## 3. Integration Points
- **RuleGenerator**: Must produce this structured JSON instead of markdown bullets.
- **Orchestrator**: Must load `active` rules from the registry and format them for prompt injection.
- **Audit**: Log state transitions (e.g., `proposed` -> `active`).

## 4. Migration Strategy
- **Dual Write**: Initially write to both `MEMORY.md` (legacy) and `memory_registry.json`.
- **Injection Switch**: Update `LangGraphOrchestrator` to read from JSON but fallback to Markdown if empty.

## Requirements Mapping
- **MEM-04**: Structured JSON registry.
- **MEM-05**: Lifecycle controls and versioning.
