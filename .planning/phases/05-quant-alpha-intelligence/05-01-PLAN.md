<objective>
Close the two spec gaps between the CONTEXT.md decisions and the current implementation of `quant_alpha_intelligence.py`, and implement the locked `{name}_{period}` result key convention.

Gap 1: RSI output must include a `state` field (`overbought`/`oversold`/`neutral`) alongside the raw value. The current implementation returns a plain float from `handle()`.

Gap 2: The CONTEXT.md defines two formal error codes — `INSUFFICIENT_DATA` (series too short) and `INVALID_INPUT` (bad params, missing required series). The current `handle()` uses `INVALID_PARAMETER` for all ValueError cases, which conflates both error types.

Gap 3: The CONTEXT.md locks the `{name}_{period}` result key convention so multi-instance requests (e.g. RSI with period=14 and RSI with period=28) produce separate labeled keys (`rsi_14`, `rsi_28`) rather than overwriting each other. The current `handle()` uses `req["name"]` as the key.

Purpose: All three gaps matter for downstream consumers. Phase 6's OrderRouter calls `calculate_indicators` for ATR; QuantModeler's system prompt instructs it to use the RSI state field; agents submitting two RSI requests for different periods depend on both results being present. Agents receiving `INVALID_PARAMETER` instead of `INSUFFICIENT_DATA` cannot distinguish a fixable data problem from a configuration error.

Output: Updated `quant_alpha_intelligence.py` and `tests/test_quant_alpha_intelligence.py` with 12 passing tests.
</objective>

CONTEXT.md specifies two distinct codes:
- INSUFFICIENT_DATA — raised when series is too short (e.g., "requires at least 15 data points")
- INVALID_INPUT — raised for bad params, missing required series, invalid values

RSI ValueError messages:
- Too short: "RSI({period}) requires at least {period+1} data points; received {len(prices)}."
- Range violation: raised in handle() before calling rsi(): "Parameter 'period'=300 is outside safe range [2, 250]."
- ATR missing series: "ATR requires 'high', 'low', and 'close' series."

CONTEXT.md RSI state spec:
- RSI returns {value: float, state: str} where state is one of: "overbought" (>70), "oversold" (<30), "neutral" (30-70)
- The raw rsi() method on TechnicalIndicators stays unchanged (returns float) — state annotation happens in handle()

CONTEXT.md result key convention (locked decision):
- Result keys follow {name}_{period} convention: rsi_14, rsi_28, macd_12_26_9, bb_20
- Period-parameterized indicators (rsi, macd, bb, atr): key = f"{name}_{params.get('period', '')}"
  - For indicators without a period param, fall back to just name
- This allows multiple instances of the same indicator with different periods to coexist in one result dict

1. INSUFFICIENT_DATA error code: In the except ValueError block, classify the error by inspecting the message string. If the message contains "requires at least" it is INSUFFICIENT_DATA. All other ValueErrors are INVALID_INPUT (replacing INVALID_PARAMETER). Example:

```python
except ValueError as e:
    msg = str(e)

```

Apply this only in handle(), not in TechnicalIndicators.rsi(). The rsi() method continues to return float or List[float] — handle() post-processes it:
```python
result = INDICATOR_REGISTRY[name](calculator, series, params, full_series)
# Annotate RSI with machine-readable state
if name == "rsi" and not full_series:
    rsi_val = result
    if rsi_val > 70:
        rsi_state = "overbought"
    elif rsi_val < 30:
        rsi_state = "oversold"
    else:
        rsi_state = "neutral"
    result = {"value": rsi_val, "state": rsi_state}
results[key] = result
```

```python
period = params.get("period", "")
key = f"{name}_{period}" if period else name
```
Replace all uses of `results[name]` with `results[key]` throughout the handle() function for the success path and the error path.
