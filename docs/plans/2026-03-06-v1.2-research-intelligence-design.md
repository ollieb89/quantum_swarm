## Phase 8: Portfolio Governance Layer

### Goal

Every trade the swarm submits is evaluated against aggregate portfolio constraints before execution. A normalized risk score is computed and persisted. No trade bypasses portfolio-level limits.

### Problem being solved

A system with perfect per-trade stops (Phase 6) is still institutionally unsafe if aggregate exposure is unmanaged. Five positions in correlated assets can pass all per-trade checks while representing one concentrated bet.

### Components

#### 8.1 Portfolio Exposure Gate

Hard limits enforced in `OrderRouter` before any venue call:

| Constraint | Default Policy |
|---|---|
| Max notional exposure | 50% of total capital |
| Single-asset concentration | 20% of total capital |
| Max concurrent open trades | Configurable (e.g. 10) |

Every rejection persists a structured reason code:

```json
{
  "rejection_code": "PORTFOLIO_MAX_NOTIONAL_EXCEEDED",
  "portfolio_heat": 0.34,
  "requested_notional": 12000.0,
  "post_trade_notional_ratio": 0.56
}
```

#### 8.2 Portfolio Heat Definition

Portfolio heat is formally defined as stop-adjusted worst-case loss, not gross notional:

```
portfolio_heat = sum(max_loss_per_position) / total_capital

max_loss_per_position = abs(entry_price - stop_loss_level) * position_size
```

This ties heat directly to downside risk rather than exposure size.

#### 8.3 Pre-Trade Risk Scoring Engine

A normalized 0.0-1.0 risk score computed before every trade, persisted to the audit record:

```
risk_score = weighted_sum(
    stop_distance_score * 0.40,   # hard market fact
    portfolio_heat_score * 0.40,  # hard market fact
    agent_confidence_score * 0.20 # soft signal, lower weight
)
```

Agent confidence is weighted lowest to avoid noisy calibration dominating the score.

**Hard gates vs soft scoring boundary** (must be maintained):
- Hard limits reject trades unconditionally
- Risk score ranks/warns but never silently overrides policy

#### 8.4 Schema Extensions

New columns added to the PostgreSQL trade warehouse:

```sql
ALTER TABLE trades ADD COLUMN trade_risk_score FLOAT;
ALTER TABLE trades ADD COLUMN portfolio_heat FLOAT;
ALTER TABLE trades ADD COLUMN rejection_code VARCHAR(64);
ALTER TABLE trades ADD COLUMN rejection_detail JSONB;
CREATE INDEX idx_trades_exit_time ON trades(exit_time) WHERE exit_time IS NOT NULL;
```

#### 8.5 Stop Hardening (prerequisite before Phase 8 gate)

Before the portfolio gate is implemented, complete these hardening checks from Phase 6:

1. **Stop direction validation**: long orders require `stop_loss_level < entry_price`; short orders require `stop_loss_level > entry_price`
2. **Stop quality validation**: numeric, finite, non-null, non-zero, non-negative, not absurdly close unless flagged
3. **Stop provenance persistence**: ATR at entry, ATR multiplier used, computed risk distance

### Success Criteria

1. A trade that would push portfolio notional above 50% of capital is rejected with `PORTFOLIO_MAX_NOTIONAL_EXCEEDED` before any venue call
2. A trade that would push single-asset concentration above 20% is rejected with `PORTFOLIO_CONCENTRATION_EXCEEDED`
3. Every executed trade has `trade_risk_score` and `portfolio_heat` populated in the PostgreSQL audit record
4. Every rejected trade has a structured `rejection_code` and `rejection_detail` persisted
5. Stop direction and quality are validated before the portfolio gate is reached

### Tests (minimum)

- Exposure limit rejection
- Single-asset concentration rejection
- Risk scoring logic with known inputs
- Stop direction validation (long and short)
- Stop quality validation (null, zero, wrong direction)

### Future hardening (Phase 8+)

- Directional aggregation: group exposure by correlated clusters, not just individual assets
- Sector/theme concentration limits
- Long/short bias limits

## Phase 9: Structured Memory Registry

### Goal

Replace freeform `MEMORY.md` appends with a governed, machine-readable rule system. Every memory-driven behavioral change is traceable to a specific structured rule with evidence, status, and version. No rule influences live trading without human approval.

### Problem being solved

Phase 7 (Self-Improvement Loop) produces advisory reports with proposed rules. If those rules are applied as freeform text appended to `MEMORY.md`, they are ungoverned: no versioning, no evidence, no lifecycle control, no audit trail.

### Rule Schema

```json
{
  "id": "mem_0001",
  "title": "Widen stops in high-volatility trend regimes",
  "type": "risk_adjustment",
  "scope": "trade_entry",
  "condition": {
    "regime": "high_volatility_trend"
  },
  "action": {
    "parameter": "atr_multiplier",
    "from": 2.0,
    "to": 2.5
  },
  "evidence": {
    "sample_size": 41,
    "window_start": "2026-01-10",
    "window_end": "2026-03-01",
    "summary": "2.0 ATR stops were hit too frequently before trend continuation."
  },
  "status": "proposed",
  "version": 1,
  "created_at": "2026-03-06T12:00:00Z",
  "reviewed_at": null,
  "approved_by": null,
  "expires_at": null
}
```

### Rule Lifecycle

```
proposed -> validated -> active -> deprecated
                      -> rejected
```

| Status | Meaning |
|---|---|
| `proposed` | Generated by ImprovementAnalyzer, awaiting review |
| `validated` | Passes schema + evidence checks, ready for human approval |
| `active` | Human-approved, injected into agents |
| `deprecated` | Superseded by newer rule or evidence expired |
| `rejected` | Human-rejected, never applied |

**Critical control**: no rule reaches `active` without a human writing `APPROVE` back to the review artifact. The swarm cannot self-activate rules.

### Components

#### 9.1 ImprovementAnalyzer

Reads from the PostgreSQL trade warehouse:
- Stop-loss outcomes by regime
- ATR multiplier calibration (stop hit rate per multiplier value)
- Macro-vs-technical divergence correlation with P&L
- Portfolio heat at trade entry vs outcome
- Rejection code frequency analysis

All computations are **purely deterministic** — no LLM involved at this stage.

#### 9.2 RefinementRuleGenerator

Maps metric thresholds to typed rule templates. Examples:

- `stop_hit_rate_HIGH_VOL > 0.6` → `CAUTION: ATR multiplier too tight in HIGH_VOL`
- `macro_technical_divergence_loss_rate > 0.7` → `AVOID: entries where macro and technical direction diverge`
- `portfolio_heat_at_loss_mean > 0.3` → `PREFER: reduce position size when portfolio heat > 0.25`

Rule logic and parameters are deterministic. LLM may optionally draft the human-readable `evidence.summary` prose, but cannot modify rule type, scope, condition, or action.

#### 9.3 Memory Storage

Rules stored in `data/memory/rules.json`. `MEMORY.md` becomes a human-readable view generated from `rules.json`, not the source of truth.

#### 9.4 Agent Injection

At session start, the skill registry loads only rules with `status: active` or `status: validated`. Agents receive injected context filtered to their scope (`trade_entry`, `risk_assessment`, `regime_detection`, etc.).

#### 9.5 Audit Trail

Every rule state transition is logged:
```json
{
  "rule_id": "mem_0001",
  "from_status": "proposed",
  "to_status": "active",
  "actor": "human",
  "timestamp": "2026-03-06T14:00:00Z"
}
```

### Success Criteria

1. `RefinementRuleGenerator` produces at least one typed rule from a batch of trade warehouse records
2. A proposed rule with no human approval cannot reach `active` status
3. An `active` rule is injected into the correct agent scopes at session start
4. A `deprecated` or `rejected` rule is not injected
5. Every state transition is written to the audit log with actor and timestamp
6. `MEMORY.md` can be regenerated from `rules.json` deterministically

### Tests (minimum)

- Rule generation from known metric inputs
- Lifecycle state machine (valid and invalid transitions)
- Agent injection filtered to correct scope and status
- Audit trail written on each state transition
- `MEMORY.md` regeneration from `rules.json`

## Phase 10: Analyst Intelligence Upgrade (intent-level)

**Research source**: TradingAgents (arXiv:2412.20138) + virattt/ai-hedge-fund + FinRobot

### Intent

Upgrade the L2 analyst layer using production-tested prompt templates and LLM routing from peer implementations.

### Planned changes

1. **Prompt templates**: Replace existing MacroAnalyst and QuantModeler system prompts with TradingAgents-derived role definitions for Fundamentals, Sentiment, News, and Technical analysts
2. **Risk team triad**: Add Aggressive/Neutral/Conservative risk agent variants (TradingAgents pattern) to supplement existing weighted consensus gate
3. **LLM routing**: Gemini Flash for data retrieval nodes; Gemini Pro (or equivalent deeper model) for analysis and debate nodes
4. **Structured JSON handoffs**: Adopt AutoHedge-inspired structured JSON output format for all L2->L3 agent handoffs (replaces freeform text signals)
5. **Indicator expansion**: ADX (trend strength), VWAP, Keltner Channels added to the indicator registry (Phase 5 registry pattern makes this straightforward)

### Depends on

Phase 9 (structured memory must be in place before analyst upgrade, so improved signals are traceable to governed rules)

## Phase 11: RL Executor — Full Closed Loop (intent-level)

**Research source**: FinRL (AI4Finance-Foundation, 10k+ stars), NautilusTrader RL compatibility docs

### Intent

Wrap the NautilusTrader backtester as a FinRL gym environment, train a PPO policy, and register the trained model as a live L3 executor that the L1 orchestrator can invoke alongside existing rule-based executors.

### Planned architecture

```
NautilusTrader backtester
  wrapped as
FinRL gym environment (TradingEnv)
  training input to
PPO policy (Stable-Baselines3)
  policy checkpoint saved as
RLExecutor (L3 stateless executor)
  registered in
L1 skill registry -> invocable by orchestrator
```

### Training pipeline (FinRL-Trading pattern)

```
train split -> PPO training -> checkpoint
test split  -> out-of-sample validation
live        -> paper trading with human review before capital allocation
```

### Depends on

Phase 10 (analyst upgrades produce higher-quality structured signals for the RL agent's observation space)

### Key risk

The NautilusTrader<>FinRL bridge is the novel engineering work — no existing project does both. Budget for a discovery spike before committing to full implementation.

## Architecture Invariants (all phases)

These must not be violated by any v1.2 work:

1. **Lazy LLM init**: all module-level LLM instances use getter pattern (`_get_llm()`) — `GOOGLE_API_KEY` not available at import
2. **No auto-activation of memory rules**: human approval required before `active` status
3. **Hard gates before soft scoring**: portfolio limits are binary reject/pass; risk scores advise but do not override
4. **Wilder's smoothing for RSI and ATR**: not simple averages
5. **8-decimal precision**: round only at output serialization boundary
6. **psycopg3 async throughout**: not psycopg2
7. **`asyncio.run()` in tests**: not `asyncio.get_event_loop().run_until_complete()`
