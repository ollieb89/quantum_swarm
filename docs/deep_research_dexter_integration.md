## 2. Integration Architecture: "Specialist-as-a-Service"

Because the swarm is Python-based and Dexter is TypeScript-based, we will use a **Decoupled Invocation Pattern**.

### 2.1 The "Deep Researcher" Node
We propose adding a new L2 node in `src/graph/orchestrator.py`:

```python
def deep_research_node(state: SwarmState):
    """Invokes Dexter for deep fundamental analysis."""
    ticker = state.get("ticker")
    query = f"Perform a deep research and DCF valuation for {ticker}."
    
    # Execution via Bun CLI (or future MCP)
    result = run_dexter_cli(query)
    
    return {
        "deep_research_report": result,
        "messages": [AIMessage(content=f"Dexter: Deep research completed for {ticker}.", name="Dexter")]
    }
```

## 3. Implementation Path

### Phase 1: CLI Wrapper (Immediate)
Utilize Dexter's existing CLI interface via Python `subprocess`.
*   **Command**: `bun run src/index.tsx --query "[QUERY]"`
*   **Output**: Capture STDOUT and parse the Markdown "Final Answer".

### Phase 2: MCP Server (Scalable)
Wrap Dexter's tools in an **MCP Server** (Model Context Protocol).
*   Our LangGraph agents can then call `read_filings` or `dcf_valuation` directly as tools, even though the execution happens in the Bun environment.

## 4. Operational Requirements

### 4.1 Environment Sync
Dexter requires specific keys to be added to the project `.env`:
*   `FINANCIAL_DATASETS_API_KEY`: Fundamental and analyst data.
*   `EXASEARCH_API_KEY`: Neural search for web research.
*   `ANTHROPIC_API_KEY`: Preferred for prompt caching in long research loops.

### 4.2 Error Handling & Timeouts
Deep research tasks often take 45-90 seconds.
*   **Async Pattern**: The `deep_research_node` should support async execution to avoid blocking the swarm heartbeat.
*   **Iteration Limits**: Dexter should be capped at `maxIterations: 10` for cost control.
